```

3. 도구 실행 (recommend_projects)
   ├─ 키워드 추출 (TF-IDF)
   │  └─ 예상 소요: < 0.1초 ✅
   │
   ├─ BGE-M3 임베딩 생성
   │  ├─ 캐시 확인
   │  ├─ Ollama BGE-M3 API 호출
   │  │  └─ 예상 소요: 5~15초 ⚠️
   │  │     - 모델 로딩 (첫 호출 시): 3~10초
   │  │     - 임베딩 생성: 2~5초
   │  └─ 캐시 저장
   │
   ├─ PostgreSQL 벡터 검색
   │  └─ 예상 소요: 0.1~1초 ✅
   │     - pgvector 인덱스 사용 시 매우 빠름
   │
   └─ 매칭 키워드 찾기
      └─ 예상 소요: < 0.1초 ✅

4. Qwen 2.5 Tool Calling (2차 호출)
   ├─ 도구 결과를 Qwen에 재전달
   ├─ Ollama Qwen 2.5 API 호출
   │  └─ 예상 소요: 5~20초 ⚠️
   │     - 추론 시간: 5~20초
   └─ 최종 응답 생성

5. FastAPI → 웹 UI 응답
   └─ 소요 시간: < 0.1초 (네트워크)
```

## 🔍 주요 병목 지점 분석

### 1. Qwen 2.5 Tool Calling (1차 호출) - **예상 10~30초** ⚠️⚠️⚠️

**위치**: `server/app.py:274-290`

```python
# 1단계: Qwen 호출 (도구 정의 포함)
resp = await client.post(
    "/api/chat",
    json={
        "model": QWEN_MODEL,
        "messages": messages,
        "tools": QWEN_TOOLS,  # 도구 정의 포함
        "stream": False,
        "options": {"temperature": 0.7, "num_predict": 1024},
    },
)
```

**소요 시간이 긴 이유**:
- **모델 로딩**: Qwen 2.5 모델이 메모리에 없으면 디스크에서 로딩 (5~15초)
- **도구 정의 처리**: `tools` 파라미터에 도구 정의가 포함되어 있어 추론 시간 증가
- **긴 컨텍스트**: `num_predict: 1024`로 최대 1024 토큰 생성
- **Tool Calling 추론**: 도구를 호출할지 판단하는 추가 추론 시간

**예상 소요 시간**:
- 첫 호출 (모델 로딩): 15~30초
- 이후 호출: 10~20초

---

### 2. BGE-M3 임베딩 생성 - **예상 5~15초** ⚠️⚠️

**위치**: `src/vector_embedder.py:164-196`

```python
resp = await self._client.post(
    "/api/embeddings",
    json={"model": self.MODEL_NAME, "prompt": processed_text},
)
```

**소요 시간이 긴 이유**:
- **모델 로딩**: BGE-M3 모델이 메모리에 없으면 디스크에서 로딩 (3~10초)
- **임베딩 생성**: 1024차원 벡터 생성 (2~5초)
- **캐시 미스**: 동일한 프롬프트가 캐시에 없으면 Ollama 재호출

**예상 소요 시간**:
- 첫 호출 (모델 로딩): 8~15초
- 이후 호출 (캐시 미스): 2~5초
- 캐시 히트: < 0.01초 ✅

**최적화 상태**:
- ✅ LRU 캐시 적용 (200개)
- ✅ httpx.AsyncClient 연결 재사용
- ✅ 모델 워밍업 (서버 시작 시)

### 3. Qwen 2.5 Tool Calling (2차 호출) - **예상 5~20초** ⚠️⚠️

**위치**: `server/app.py:324-338`


**소요 시간이 긴 이유**:
- **긴 컨텍스트**: 도구 결과(프로젝트 리스트 JSON)가 메시지에 포함되어 컨텍스트가 길어짐
- **추론 시간**: `num_predict: 512`로 최대 512 토큰 생성
- **모델이 이미 로딩되어 있어도**: 추론 자체가 시간 소요

**예상 소요 시간**:
- 5~20초 (컨텍스트 길이에 따라 변동)

## 🎯 100초 소요 원인 추정

### 가능한 원인들

#### 1. **Qwen 2.5 모델 로딩 지연** (가능성 높음) ⚠️⚠️⚠️

**증상**:
- 첫 호출 시 매우 느림
- 이후 호출도 느림 (모델이 메모리에서 해제되었을 수 있음)

**원인**:
- Qwen 2.5 모델 크기가 큼 (7B 파라미터)
- 디스크 I/O가 느림 (HDD 사용 시)
- 메모리 부족으로 모델이 메모리에서 해제됨

**확인 방법**:
- 서버 로그에서 `[TIMING] Qwen 1차 호출` 시간 확인
- Ollama 로그에서 모델 로딩 시간 확인

---

#### 2. **BGE-M3 모델 로딩 지연** (가능성 중간) ⚠️⚠️

**증상**:
- 임베딩 생성이 매우 느림
- 캐시 히트가 아닌 경우

**원인**:
- BGE-M3 모델 크기가 큼
- 디스크 I/O가 느림
- 모델이 메모리에서 해제됨

**확인 방법**:
- 서버 로그에서 임베딩 생성 시간 확인
- `vector_embedder.py`에 타이밍 로그 추가 필요

---

#### 3. **Qwen 2.5 추론 시간 지연** (가능성 중간) ⚠️⚠️

**증상**:
- 모델은 로딩되었지만 추론이 느림

**원인**:
- CPU만 사용 중 (GPU 미사용)
- `num_predict` 값이 큼 (1024, 512)
- 컨텍스트가 길어서 추론 시간 증가

**확인 방법**:
- Ollama가 GPU를 사용하는지 확인
- `num_predict` 값을 줄여서 테스트

---

#### 4. **네트워크 지연** (가능성 낮음) ⚠️

**증상**:
- Ollama 서버와의 통신이 느림

**원인**:
- 네트워크 대역폭 부족
- Ollama 서버가 원격에 있음

**확인 방법**:
- `localhost:11434`로 직접 접속하는지 확인
- 네트워크 지연 측정

---

#### 5. **PostgreSQL 벡터 검색 지연** (가능성 낮음) ⚠️

**증상**:
- 벡터 검색이 느림

**원인**:
- `embedding` 컬럼에 인덱스가 없음
- 데이터가 매우 많음 (수만 건 이상)

**확인 방법**:
- 인덱스 존재 여부 확인: `\d cases` (psql)
- 쿼리 실행 계획 확인: `EXPLAIN ANALYZE`

## 💡 최적화 방향 (향후 개선 시)

### 1. Qwen 2.5 호출 최적화
- 모델을 항상 메모리에 유지 (Ollama 설정)
- `num_predict` 값 감소 (512 → 256)
- Tool Calling 대신 Python 의도 감지로 우선 처리

### 2. BGE-M3 임베딩 최적화
- 캐시 히트율 향상 (캐시 크기 증가)
- 배치 임베딩 (여러 텍스트 동시 처리)
- 모델을 항상 메모리에 유지

### 3. 병렬 처리
- Qwen 2.5 (1차)와 BGE-M3 임베딩을 병렬 실행
- 도구 실행과 Qwen 2.5 (2차)를 병렬 고려

### 4. Fallback 경로 활용
- Python 의도 감지로 우선 처리 (Qwen 호출 생략)
- Qwen Tool Calling은 선택적으로만 사용
